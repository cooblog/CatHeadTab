## Favicon 代理 + 本地缓存方案

### 背景
原来所有网站图标都依赖 `s2.googleusercontent.com` 的 Google Favicon 服务，国内无法访问。

### 实施内容

**后端新增：**
- `backend/internal/handler/favicon.go` — Favicon 代理 API handler
  - 路由：`GET /api/v1/favicon?domain=xxx&sz=64`（公开接口，无需认证）
  - 多源抓取策略：Google S2 → 网站 /favicon.ico → 解析 HTML `<link rel="icon">`
  - 磁盘缓存到 `data/favicon_cache/`，SHA256 作为文件名
  - 响应头设置 `Cache-Control: max-age=31536000, immutable`（1年强缓存）
  - 失败时返回 1x1 透明 PNG

**前端新增：**
- `frontend/src/utils/favicon.ts` — 统一 Favicon 工具函数
  - `getSmartFaviconUrl()` — 智能选择：有服务器用代理，无服务器降级 Google S2
  - `getFaviconUrlAsync()` — 异步版本，支持 IndexedDB 永久缓存
  - IndexedDB 缓存（`catheadtab-favicons` 数据库），永不过期
  - 内存 Blob URL 缓存（页面生命周期内）
  - 防重复请求（pendingFetches Map）

**前端替换：**
- Desktop.tsx — 2 处（文件夹缩略图 + 链接主图标）
- Bookmarks.tsx — 1 处
- BookmarkBrowser.tsx — 2 处
- ExploreWorld.tsx — 1 处
- AddItemModal.tsx — 1 处

### 缓存层级
1. 浏览器内存 Blob URL（最快，页面生命周期）
2. IndexedDB Blob 缓存（永久，跨会话）
3. 浏览器 HTTP 缓存（后端设置 1 年强缓存）
4. 后端磁盘缓存（`data/favicon_cache/`）
5. 后端多源抓取（Google S2 / 直接 /favicon.ico / HTML 解析）

- 控制台 CORS 错误实际是由组件崩溃连锁引发的
- 已全部修复

## Bug 修复：getFaviconUrl 同步调用触发冗余 fetch 导致 CORS 错误
- 问题：`getFaviconUrl()` 被 `<img src=...>` 同步调用时，除了返回代理 URL 给 `<img>` 标签（不受 CORS 限制），还会在后台用 `fetch()` 请求同一 URL（受 CORS 限制），导致大量 CORS 报错
- 根因：`loadFaviconAsync` 内部调用了 `fetchAndCache`，而后者使用 `fetch()` API
- 修复：将 `loadFaviconAsync` 拆分为 `loadFromIndexedDB`（仅查 IndexedDB 缓存，不发网络请求），`getFaviconUrl` 同步路径不再触发 fetch
- 新增 `cacheImageFromElement` 导出函数（可选），供 `<img onLoad>` 回调使用将已加载图片缓存到 IndexedDB
- `getFaviconUrlAsync` 保留 fetch 能力，供显式异步场景使用
