## 书签操作按钮 hover 效果二次修复
- 问题：上次修复后，hover 整行仍然无法正确显示操作按钮
- 原因：Tailwind 的 `group/group-hover` 机制在该场景下不可靠，可能受到外层其他 `group` 类的干扰
- 修复方案：改用纯 CSS 实现 hover 效果
  - 行容器使用 `.bookmark-row` 类名替代 `group`
  - 操作按钮容器使用 `.bookmark-actions` 类名，默认 `opacity: 0`
  - 标题使用 `.bookmark-title` 类名
  - 在 `index.css` 中添加 `.bookmark-row:hover .bookmark-actions { opacity: 1 }` 和 `.bookmark-row:hover .bookmark-title { color: white }` 规则
  - 完全不依赖 Tailwind 的 `group-hover`，用原生 CSS `:hover` 确保可靠性

## 书签操作按钮 hover 效果三次修复（真正根因）
- 问题：hover 仍然只在图标位置触发，文字区域无反应
- 根因：Browser Body 区域第 123 行有 `<div className="absolute inset-0 border-t border-white/5" />`，这是一个全尺寸绝对定位的装饰性边框 div，虽然透明但会拦截鼠标事件，导致底层内容无法接收到 hover
- 修复：给该 div 添加 `pointer-events-none`，让鼠标事件穿透它

## 修复拖拽图标到文件夹的 Bug
- **文件**：`frontend/src/pages/Desktop.tsx`
- **根因**：`@dnd-kit/sortable` 的 `closestCenter` 碰撞检测在网格排序中，`over` 会随排序动画在文件夹和相邻元素间抖动，导致 500ms hover 定时器永远达不到
- **修复方案**：
  1. 自定义碰撞检测 `folderAwareCollision`：优先使用 `pointerWithin` 检测指针是否在文件夹矩形内，命中则直接返回文件夹作为目标；否则回退到 `closestCenter`
  2. 增加 `lastFolderOverRef` / `lastFolderOverTimeRef` 追踪最近悬停的文件夹，短暂抖动（<150ms）不重置计时器
  3. hover 延迟从 500ms 降到 300ms
  4. `handleDragEnd` 增加备选判断：即使 `folderDropTargetId` 未设置，只要 `lastFolderOverRef` 指向当前 `over` 的文件夹也允许放入
  5. `useSortable` data 增加 `isFolder` 标记供碰撞检测使用

## 修复拖入文件夹时网格重排问题
- **文件**：`frontend/src/pages/Desktop.tsx`
- **问题**：拖拽图标到文件夹上方时，sortable 的排序逻辑仍会触发位置交换动画，文件夹图标被推开滑动，而不是固定在原位等待接收
- **修复方案**：
  1. 新增 `isFolderDropPending` 状态，当 `handleDragOver` 检测到 over 目标是文件夹时设为 `true`
  2. `SortableDesktopIcon` 新增 `isFolderDropPending` prop，当为 `true` 且非拖拽中的图标，忽略 sortable 的 `transform`（设为 `undefined`）并禁用 `transition`
  3. 这样当拖拽目标是文件夹时，整个网格冻结不动，文件夹图标固定在原位
